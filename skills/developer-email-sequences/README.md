# developer-email-sequences

**Email that developers actually read** — onboarding, updates, and re-engagement without the spam.

## What It Does

Helps you create email sequences that respect developers' time and intelligence. Covers onboarding drips, changelog updates, re-engagement campaigns, and transactional emails with templates and frequency guidelines.

## When to Use

- Setting up onboarding emails for new signups
- Creating changelog/product update communications
- Building re-engagement sequences for inactive users
- Auditing your current email strategy
- When you say "developer emails," "onboarding sequence," "email drip," or "re-engagement"

## Quick Start

```
/developer-email-sequences

Create an onboarding email sequence for our API product.
We want to get developers to their first successful API call.
```

## Email Types Covered

| Type | Purpose | Max Frequency |
|------|---------|---------------|
| Onboarding | First value moment | 4 emails / 7 days |
| Activation | Turn trials into users | After first success |
| Changelog | Product updates | 1x/week max |
| Re-engagement | Win back inactive users | 4 emails / 60 days |
| Transactional | Receipts, alerts, security | As needed |

## Key Principles

1. **Every email must provide immediate value**
2. **Code snippets > marketing copy**
3. **Stop sequences when goals are met**
4. **Plain text often outperforms HTML**
5. **One CTA per email**
